# 协作约定 — 怎么跟这个用户干活

> 本文件是**可迁移的协作规则库**（工具无关，git 受控）。由 auto-memory 里的 `feedback_*` 归并而来，
> 是 AI 助手与本项目用户协作的行为准则。**新增/修订协作规则时，这里是唯一真相源**——
> 见文末「维护约定」。技术架构铁律在 [`CLAUDE.md`](../../CLAUDE.md)，此处只管"工作方式"。

---

## A. 沟通与语言

- **始终用简体中文**回答（即使代码/报错/文档是英文，解释也用中文）。不要用日文。

## B. 工作方式

### B1. 涉及方案选择必须先问用户，不替他拍板
设置页位置 / API 默认选哪个 / UI 入口形式 / 数据形态等**有多种合理选择**的决策，给 A/B/C 方案 + 各自利弊，等用户拍板再动手。（CLAUDE.md 方法论 #3）

### B2. 相关开发一次做完整，别拆碎
一个功能涉及的强相关改动（enum/持久化/UI/算法联动/守卫等）作为**一个完整闭环一次做完**，不要先做轻的、把重的拆成"后续增量"延后。
- **Why**：用户要完整可用的功能；零碎增量留半成品、反复返工、逼他多次介入验收。
- **例外**：真正独立、可延后不影响主功能可用性的（内容工程、纯打磨）才另排。
- **反例（踩过）**：Sprint 5 StudyPhase 我建议"交错策略留后续增量" → 被否，要求全做完。

### B3. 不把阶段性决定写成永久结论
描述项目定位/路线图/商业模式/开源策略时，写"**当前阶段** X"而非"永远不 X"。
- **Why**：曾把"当前阶段不做商业化准备"概括成"不做商业化考量" → 误导未来方案选择。
- **自查触发词**：永远、绝对、不会、必须、铁律——每出现一次核对用户原话；只是阶段决定就降级语气。
- 不确定的长期路线**保留为开放问题**（"待用户后期决定"），不替用户拍板；未经明确表态不写进记忆当结论。

### B4. TODO / 阶段文档要具体、可核查，半拉项必须登记 🔥
- 每条任务标现状（🔴未做/🟡半成/✅）+ 已落地部分（commit/文件）+ 还差什么 + 验收标准，**不写笼统乐观勾**。
- **任何 占位 / MVP / "后续 Task" / 回落 / 自动通过 / 暂未实装，一旦产生或读到，当场记进 TODO 或改进点池**（写清 file:line + 还差什么）。否则换个对话不逐行扫代码就**永远漏掉**。
- **Why**：本阶段靠用户直觉追问才逐次挖出 C5 整卡、七步 Step2/5 占位、Step7 迷你卡、复习默写 hint 未实装等 7 块债——全是文档没记明白导致的。
- 答"某阶段/功能是否完成"前，**先扫代码**的 `未实装/占位/fallback/延后/TODO/回落` 标记 + 卡型面板 when 分支，别信文档乐观勾。

### B5. 外部 SDK/API/定价先查官方页，不靠训练记忆
写接入代码或"免费额度/价格"声明前先 WebFetch 官方文档/价格页；产品边界（"是否支持 X"）优先做最小可执行测试。（CLAUDE.md 方法论 #1，踩过 ML Kit 不支持数学公式 / Mathpix 无持续免费层）

### B6. UI/渲染 bug 修 2 轮无果 → 改诊断驱动
同一渲染 bug 修 2 轮没好转，立刻停止猜测，在出问题的层插诊断输出（角标/Toast/Logcat，让用户一张截图就能给），用数据反推根因。（CLAUDE.md 方法论 #2）

## C. 产品铁律

### C1. APP 永远不做同日二次推送
不在用户当日学习/复习会话结束后**当天**再次推送/提醒同一条公式。
- **Why**：用户 2026-05-19 拍板——被动间隔推送（Pimsleur 式 10min/30min 队列）切碎专注、把 APP 从"用户主动使用"恶化为"索要注意力"。
- **How**：任何"会话后再推一次"设计直接否决；学习会话在用户**单次坐下时间内自闭环**（靠七步 Step7 会话内 mini-card）；跨日复习走标准 FSRS（最早次日刷新整点）；唯一允许的主动提醒是 `DailyReminderWorker` 每日单次"今日复习时间到"。写 PRD/RFC/TODO 见到"X 分钟后/当日晚些再唤起"字样立刻停下确认。

## D. 真机测试分工

- **简单操作流程尽量让用户自己点**（走 UI、点 chip、看渲染）——他想亲自体验、发现体感问题。
- Claude 只在两种情况用 adb/uiautomator 自动化：① 需改后台数据（DB 注入/构造特定状态，见 [环境与工具.md](环境与工具.md) 的 DB pull-modify-push）② 用户明确请求。
- 验收前先分工：问"这段你自己点还是我来"，默认简单流程交用户、我在旁看截图/查库佐证。
