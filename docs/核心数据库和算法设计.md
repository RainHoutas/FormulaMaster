### 一、 数据库结构设计 (Room Schema)

为了保证“公式内容的更新”绝不会覆盖或影响“用户的学习进度”，必须将**静态资产（公式本身）\**与\**动态状态（用户的学习记录）\**严格分离，并增加\**日志流水表**以支撑热力图统计。

建议在 Android Studio 中建立以下三个核心 Entity：

#### 1. 静态公式库表 (`Formula_Entity`)

这张表的数据是随 App 打包预置的（可以通过读取本地 JSON 填充），只读或仅在版本更新时全量覆盖。

Kotlin

```
@Entity(tableName = "formulas")
data class FormulaEntity(
    @PrimaryKey val formulaId: String,          // 唯一标识符，如 "calc_limit_01"
    val subject: String,                        // 科目：高数、线代、概率论
    val chapter: String,                        // 章节：如 "一元函数微积分"
    val title: String,                          // 公式名称：如 "洛必达法则"
    val latexCode: String,                      // 完整的 LaTeX 源码
    val clozeData: String,                      // JSON 字符串：填空题的挖空坐标/片段
    val derivationSteps: String,                // JSON 字符串：推导过程片段数组
    val tags: String, //新增字段：应用场景标签（如：求极限、级数展开、中值定理应用）在数据库中建议以 JSON 字符串或逗号分隔存储
    val difficultyLevel: Int                  // 客观难度评级 (1-5)，用于初次下发时的权重
)
```

**附：`clozeData` 字段 JSON 结构示例约束：**

```json
[
  {
    "index": 1, 
    "placeholder": "\\frac{1}{n!}", 
    "options": ["\\frac{1}{n!}", "\\frac{1}{(n+1)!}", "n!"]
  }
]
```

#### 2. 用户学习状态表 (`Study_State_Entity`)

这是最核心的状态表。采用 **FSRS (Free Spaced Repetition Scheduler)** 算法的变体参数，比传统 SM-2 更适合短期高强度的考研冲刺。

Kotlin

```
@Entity(
    tableName = "study_states",
    foreignKeys = [ForeignKey(
        entity = FormulaEntity::class,
        parentColumns = ["formulaId"],
        childColumns = ["formulaId"],
        onDelete = ForeignKey.CASCADE // 公式若被官方删除，学习记录级联删除
    )]
)
data class StudyStateEntity(
    @PrimaryKey val formulaId: String,
    
    // 状态机枚举：0=未学习(New), 1=记忆中(Learning), 2=复习中(Reviewing), 3=已掌握(Mastered)
    val learningState: Int, 
    
    // FSRS 核心算法字段
    val stability: Double,      // 记忆稳定性 (S)：决定复习间隔的天数
    val difficulty: Double,     // 公式主观难度 (D)：决定 S 的增长速度
    
    // 调度时间戳
    val lastReviewTime: Long,   // 上次复习的 Unix 时间戳
    val nextReviewTime: Long,   // 下次安排复习的 Unix 时间戳
    
    // 统计字段（用于动态调整和考试阶段）
    val totalReviews: Int,      // 总复习次数
    val lapses: Int             // 遗忘次数（在复习或考试中选了“忘记”的次数）
)
```

#### 3. 学习流水日志表 (`Review_Log_Entity`)

**（超额完成项）**：绝大多数独立开发者会忽略这张表，但这对于实现“类似 GitHub 的每日学习热力图日历”和“阶段性考试学习情况溯源分析”至关重要。

Kotlin

```
@Entity(tableName = "review_logs")
data class ReviewLogEntity(
    @PrimaryKey(autoGenerate = true) val logId: Long = 0,
    val formulaId: String,
    val reviewTime: Long,       // 产生记录的时间戳
    val interactionType: Int,   // 交互类型：1=初次记忆探索, 2=日常复习唤醒, 3=阶段考试严格默写
    val userRating: Int,        // 用户反馈：1=忘记/错误, 2=困难, 3=顺利, 4=极易
    val costTimeMs: Long        // 该题停留耗时（毫秒），耗时过长说明实际上不够熟练
)
```

### 二、 核心算法与状态机流转设计

你的计划中明确了“记忆、复习、测试”三个阶段。我们需要用一个严格的**状态机（State Machine）\**和\**数学模型**来驱动它，确保“考得好的极大减少复习，考得差的加强复习”。

#### 1. 生命周期状态机 (Life-cycle State Machine)

- **状态 `0 (New)` -> `1 (Learning)`：** * **触发：** 用户主动在“记忆”模块点击探索公式。
  - **动作：** 初始化算法参数。D (难度) 继承公式的客观难度，S (稳定性) 设为 1 天。下次复习时间设为明日。
- **状态 `1 (Learning)` ↔ `2 (Reviewing)`：**
  - **触发：** 每日“复习”模块。
  - **动作：** 每日调度 `nextReviewTime <= 当天` 的公式。若连续 3 次评分 ≥3（顺利），状态迁入 `2`。若在状态 `2` 中评分 =1（忘记），退回状态 `1`。
- **状态 `2 (Reviewing)` -> `3 (Mastered)`：**
  - **触发：** 系统判定。当计算出的间隔 S>30 天，自动标记为“已掌握”，不再出现在每日复习流中，**仅在“测试”模块中出现**。
- **状态 `3 (Mastered)` -> `1 (Learning)` (惩罚降级)：**
  - **触发：** 在“阶段性考试”模块中严格测试出错。
  - **动作：** 触发严厉的遗忘惩罚（Lapse Penalty）。公式直接被打回 `Learning` 状态，并且主观难度 D 急剧上升。

#### 2. 本地复习统筹算法 (基于间隔乘数与主观难度)

考虑到手机端本地运算的性能和逻辑的清晰度，推荐使用以下改进版的调度公式计算下一次复习间隔（以天为单位）。

每次用户完成复习或测试（给出评分 R∈{1,2,3,4}）后，执行以下核心计算：

**步骤一：更新主观难度 (D)** 根据用户的反馈评分 R 更新公式难度。连续选简单，难度下降；选忘记，难度急剧上升。公式如下：

Dnew=Dold+α⋅(3−R)

*(注：α 为难度调节步长，建议设为 0.5。限制 D 的范围在 [1.0,5.0] 之间)*

**步骤二：计算稳定性/间隔 (S)** 新间隔的计算取决于旧间隔以及当前的记忆难度。

- 如果 R=1 (忘记/错误)：触发遗忘逻辑，间隔重置，但保留一定基础。

  Snew=min(1.0, Sold×0.2)

- 如果 R>1 (记住)：间隔时间扩大。公式越简单（D 越小），评分越高（R 越大），跨度乘数越大。

  Snew=Sold×(1+DnewR−1)

**步骤三：考试测试的特殊修正权重** 在“考试”模式下（无提示的严格默写）：

- 如果完全正确（评分视同 R=4）：由于是无提示的硬核提取，记忆效果远超普通复习，此时给 Snew 额外乘以一个奖励系数（如 1.5），使得该公式直接休眠数月。
- 如果出现错误（评分视同 R=1）：不仅 Snew 重置为 1，且 `lapses`（遗忘次数）字段 +1。

**步骤四：落库更新** 将最新的时间戳写入数据库：

nextReviewTime=currentTime+(Snew×24×60×60×1000)

**步骤五：考前收敛机制 (Sprint Mode)** 当系统时间距离全局设定的“目标考试日 (Target Date)” $\le 30$ 天时，触发冲刺逻辑： 1. 强制将所有 $S > 15$ 的公式的 $S$ 值减半。 2. 即使是 `Mastered` (状态3) 的公式，也会被随机按比例抽入每日复习池，进行高频唤醒。

### 三、 为什么这套设计能超额完成目标？

1. **极强的抗干扰性与扩展性：** 通过 `Room` 数据库将公式表与用户状态表拆分。即使你明年更新了 App，修正了某个公式的 LaTeX 错别字（更新 `FormulaEntity`），用户的复习进度（`StudyStateEntity`）也完全不会丢失。
2. **避免“过度复习”陷阱：** 引入 `Mastered` 状态。传统的单词软件会让简单词汇在几年后依然出现，这对于需要大量刷题的考研数学是浪费时间。当公式达到 `Mastered` 状态后，它从日常队列中消失，完全交由低频的“阶段测试”接管。
3. **多维度数据统计赋能：** 流水日志表 (`ReviewLogEntity`) 记录了每次复习的**耗时**。如果你后期想做进阶分析，可以通过查询语句筛选出“虽然选了‘顺利’，但耗时超过 15 秒的公式”，这往往是用户的**隐性薄弱点**，你可以在系统层面强制它们进入下一次测试名单。
