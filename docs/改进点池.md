## 📥 待评估

### [交互] MathFormulaView 公式渲染居中边缘问题
- **优先级**：P2
- **日期**：2026-04-25
- **描述**：Sprint 1 Task 1.7 收尾时（基于真机诊断数据）已修复公式过小、过大、左对齐等核心问题，
  当前 `math_template.html` 使用 `100vh + clamp(20px, 55vh, 52px)` 方案在大多数场景效果良好。
  但仍有少数情况下公式**未完美垂直居中**（视觉略偏上 / 略偏下），具体场景待用户使用时收集。
- **已知触发条件（部分）**：
  - 包含特别高的纵向结构（如 `\int_{a}^{b} \frac{...}{...}` 同时带积分上下限和分式）
  - WebView 在异步加载字体后未触发 reflow，初次渲染时 `el.scrollHeight` 可能不准
- **可能的修法方向**：
  - 使用 `font-metrics-override` CSS 显式设置 KaTeX 字体的 ascent/descent
  - 切换 `place-items: center start` → 计算 KaTeX 内部 `.katex-display` 的 `baseline-shift` 后微调
  - 用 ResizeObserver 监听公式实际 bounding box 变化，动态调整 padding-top
- **验证方式**：覆盖 5 种典型公式（短表达式 / 上下标 / 分式 / 求和带界 / 积分带分式）的居中度
- **暂缓原因**：当前效果"非常可用"（用户反馈），优先推进 Sprint 1 Task 1.8/1.9 收尾
- **关联文件**：`app/src/main/assets/math_template.html`

### [合规] SimpleTex App 授权方式作为可选鉴权
- **优先级**：P3
- **日期**：2026-04-25
- **描述**：SimpleTex 提供两种鉴权：
  - **UAT**（已接入）：单 token 直传，简单
  - **App 授权**（未接入）：APP_ID + APP_SECRET，MD5 签名 + timestamp 防重放，**官方建议生产环境使用**
- **当前威胁模型分析**：UAT 已足够安全（HTTPS + Tink AES256-GCM 本地加密 + Android Keystore），
  App 授权的"防嗅探/防重放"优势对纯本地应用增量很小。详见 docs/API_Key_获取指南.md。
- **重启条件**（满足任一则纳入）：
  1. 项目加入云端同步功能，需要服务端代用户调 SimpleTex（这是官方警告的真实场景）
  2. 用户反馈 SimpleTex 因"我用了 UAT"被警告 / 限流
  3. SimpleTex 后期对 UAT 增加新限制（QPS 削减、月度上限等）
- **关联文件**：`domain/SimpleTexApiRecognizer.kt`、`data/RecognizerPreference.kt`（增加 SECRET 字段）、
  `ui/screen/SettingsScreen.kt`（鉴权方式选项）
- **实现参考**：<https://doc.simpletex.cn/zh/api/auth_app.html>

### [识别] 端侧 TFLite/ONNX 手写公式识别（L1 本地方案，聚焦考研符号集）
- **优先级**：P3
- **日期**：2026-04-25
- **描述**：ML Kit 不支持数学公式识别，但以下端侧路线技术上完全可行，且比通用数学识别更聪明：
  - **输入**：笔画时序数据 `[(x,y,t)]`，比图像 OCR 轻量，无需拍照/截图
  - **模型**：小型 LSTM 或 TCN，针对考研数学核心符号集（~150个：高数、线代、概率的常见符号和结构），
    基于 CROHME 数据集（Competition on Recognition of Online Handwritten Mathematical Expressions）训练
  - **部署**：转换为 TFLite 或 ONNX Runtime Mobile 格式，模型大小目标 10~30MB，完全离线运行
  - **识别范围**：考研数学常见符号和简单嵌套结构（如 `\int_a^b f(x)dx`、`\frac{d}{dx}`、行列式等）；
    复杂多级嵌套（如三重积分）识别率会下降，但覆盖大多数考研题型
  - **工程步骤**：处理 CROHME InkML 数据 → 训练符号分类器 → 结构解析层（上下标、分式等） → TFLite 转换 → Android 集成
- **关联文件**：待新建 `domain/TfLiteInkRecognizer.kt`；需新增 TFLite 依赖
- **主要挑战**：
  1. 目前无现成可用的 TFLite 数学公式识别模型，需自行训练（约 1-2 周，GPU 可用 Colab 免费额度）
  2. 符号识别 → LaTeX 组装需要结构解析层，工程量常被低估
  3. 本质是之前搁置的 L2 的精简版，门槛仍需要 ML 训练能力
- **启动条件**：A1/A2 API 路线稳定运行后，有意愿投入 ML 训练时再做

### [合规] ML Kit native lib 16 KB 页对齐（Android 15 上架要求）
- **优先级**：P2
- **日期**：2026-04-25
- **描述**：当前 `com.google.mlkit:digital-ink-recognition:18.1.0` 内置的
  `lib/arm64-v8a/libdigitalink.so` 未按 16 KB 边界对齐，IDE 提示：
  > APK is not compatible with 16 KB devices. Some libraries have LOAD segments
  > not aligned at 16 KB boundaries: lib/arm64-v8a/libdigitalink.so
  Google Play 政策：自 **2025-11-01** 起，所有面向 Android 15+ 的新应用与更新
  必须支持 16 KB 页大小，否则禁止提交。
- **当前影响**：本地开发与现有 4 KB 设备运行正常，**仅影响上架 Google Play 的合规性**
- **处理方向**：
  1. **首选**：等 ML Kit 团队发布支持 16 KB 的 `digital-ink-recognition` 新版本，升级依赖即可
  2. **备选**：若上架前仍未修复，需评估是否切换到不含 native lib 的实现，或暂时只在
     国内分发（不走 Google Play）
- **重启条件**：项目准备首次发布到 Google Play 前必须解决；在此之前持续跟进 ML Kit 版本

### [交互] 首次启动识别器配置引导（Onboarding）
- **优先级**：P1
- **日期**：2026-04-25
- **描述**：用户第一次打开 App 时，若尚未配置任何识别器，应弹出引导流程，而非直接落到空设置页。引导内容分两块：
  1. **本地模型（L1 ML Kit）**
     - 说明前提条件：需要 Google Play 服务（GMS）
     - GMS 存在时展示「下载数学公式模型（~4MB）」按钮

     - 下载成功：Snackbar 或 Dialog 提示「模型已就绪，可直接使用」
     - 下载失败（无网络等）：内联错误文案 + 「重试」按钮
     - GMS 缺失时：标灰该选项，附文案「需 Google Play 服务，请使用 API 识别」
  2. **云端 API（A1 Mathpix / A2 SimpleTex）**
     - 简短说明网络要求（Mathpix 需境外网络或代理；SimpleTex 国内可用）
     - 提供「如何获取 API Key」可点击链接（Mathpix: snip.mathpix.com；SimpleTex: simpletex.cn）
     - API Key 填写后展示「测试连接」按钮，点击发起最小测试请求，在按钮旁显示结果（✅ 连接正常 / ❌ Key 无效 / ❌ 网络超时）
     - 测试连接按钮在 Task 1.7 SettingsScreen 也需同步保留，方便事后重新验证
- **关联文件**：待新建 `ui/screen/OnboardingScreen.kt`；`ui/screen/SettingsScreen.kt`（Task 1.7）；`domain/MlKitInkRecognizer.kt`（下载逻辑已有）
- **备注**：Task 1.7 已规划 SettingsScreen 含测试连接按钮，本条重点是**首次启动的主动引导入口**；两者可共用配置 UI 组件，避免重复实现
