### [识别/交互] 手写识别触发策略优化（每次抬手都识别 → API 消耗过快）
- **优先级**：P2
- **日期**：2026-07-16（Sprint 6.8 真机使用发现）
- **背景**：手写作答（`TestCanvas`，`lightDebounceMs=300`）现在**每完成一笔（抬手）就触发一次 OCR 识别**。
  用户习惯抬手分笔写公式，导致一条公式被识别很多次 → **API 调用次数 / 费用飙升**（尤其接 Mathpix 按次计费）。
- **诉求**：优化整体手写交互的识别触发时机，减少无谓的中间识别。候选方向（待设计）：
  - 拉长 debounce / 停笔更久才识别；或
  - **改为显式触发**（写完点「识别」按钮才调 API，去掉自动预览）；或
  - 本地先攒笔画，仅在用户主动「提交」时识别一次（对手写作答判分场景最省——中间过程不需要预览）
- **⚠ 现状判断（用户拍板 2026-07-16）**：**当前处学习流程构建阶段，不深改此交互**；现状"能用"，先记录不着急做。
- **重启条件**：学习流程阶段收尾后 / 专门做 OCR 交互打磨时。
- **附带 UI 打磨**：`HandwrittenAnswerArea` 的手写画布区**显得空、"提交判分"按钮被顶到屏幕最底**，布局需收紧（同批打磨）。
- **关联文件**：`ui/component/TestCanvas.kt`（识别触发/debounce）、`ui/component/HandwrittenAnswerArea.kt`（作答区布局）

### [交互] 全面震动 / 触觉反馈适配
- **优先级**：P2
- **日期**：2026-05-01
- **背景**：当前震动反馈仅在严测/复习的"出错"场景触发（[`vibrateError`] 200/400ms），其余交互
  全部静默。但震动作为"已操作成功"的体感锚点，能显著降低用户对界面响应性的怀疑。
- **建议覆盖范围**：
  - **复习评分按钮（1/2/3/4）点击**：短促 confirm 震动（HapticFeedback.LongPress 等级）
  - **严测「都不对」/「提交核对」按钮**：confirm
  - **公式激活（首次进入复习队列）**：success 震动 + 可视化反馈
  - **设置页保存 / 测试连接成功**：success
  - **Onboarding 页面切换 / 完成「开始使用」**：subtle tick
  - **Pager 翻页边界（已是第一/最后一页还在拖）**：edge tick
  - **错误场景扩展**：识别失败 / 网络超时 / Snackbar error 弹出时同步震动（与 vibrateError 共用 API）
- **实现思路**：
  1. 抽出 `ui/util/Haptic.kt`（LocalHapticFeedback 包装 + 兜底 Vibrator API），区分 5 档：
     `Tick` / `Confirm` / `Success` / `Warning` / `Error`
  2. 各 Composable 通过 `LocalHapticFeedback` 取实例后按场景调用
  3. 设置页加"震动反馈"开关（默认 ON），用户可一键全局关闭
- **关联文件**：`ui/util/Haptic.kt`（新建）、`ui/component/TestCanvas.kt`、`ui/component/ReviewCard.kt`、
  `ui/screen/SettingsScreen.kt`、`ui/screen/OnboardingScreen.kt`、`data/AppPreference.kt`

### [工程] 日期时间库迁移至 kotlinx-datetime（KMP 扩展准备）
- **优先级**：P3
- **日期**：2026-04-30
- **描述**：当前 domain 层（`ReviewScheduler.truncateToRefreshHour` 等）使用 `java.time`
  （`ZonedDateTime` / `Instant` / `ZoneId`）。`java.time` 在 minSdk ≥ 26 时原生可用，是现阶段最优选。
  若未来将 domain 层扩展为 Kotlin Multiplatform（iOS / Desktop），需切换为
  `kotlinx-datetime`（JetBrains 官方 KMP 日期库），因为 `java.time` 不跨平台。
- **迁移成本估计（2026-07-13 复核，上修）**：**中**，非"低"。实际用到 `java.time` 的
  已扩散到约 10 个文件（`ReviewScheduler` / `AppPreference` / `ReviewSessionRepository` /
  `ErrorReportProcessor` / `DailyReminderWorker` + UI 的 `HeatmapCalendar` / `OnboardingScreen` /
  `SettingsScreen` / `FormulaDetailScreen` / `SettingsViewModel`）。且 `kotlinx-datetime`
  **没有** `TemporalAdjusters`（`AppPreference` 算"12 月最后一个周六"用到）和成熟的
  `DateTimeFormatter`（多处格式化用到），这些得手工重写，不是纯 import 替换。
- **⚠️ 现状判断（2026-07-13）**：当前**纯安卓，此迁移零用户收益**，唯一价值是将来上
  iOS/Desktop（KMP）时的预备。而改动踩的正是复习调度的时区/跨天核心，DST/日界正确性按
  项目铁律**必须真机验**，不宜在无设备场次盲迁。**结论：非启动 iOS/多平台不做**，保持 P3 挂起。
- **重启条件**：项目明确启动 KMP 改造（共享 domain 层到非 Android 平台）时才纳入。
- **关联文件**：`ReviewScheduler` / `AppPreference`（`TemporalAdjusters`）/ `HeatmapCalendar` /
  `OnboardingScreen` / `SettingsScreen` 等约 10 个文件（见上）

  - 推断：详情页 / 复习卡片 / Tab 切换 等"首次合成内容稍重的页面"都有类似现象
- **本条目目标**：作为 UI 全打磨阶段的性能子目标，**一次性集中诊断 + 修**所有
  页面切换 / 启动场景的可感知卡顿，不再零敲碎打。
- **可能根因**（待诊断驱动确认）：

  2. **SAF launcher 提升到 Activity 级**：在 MainActivity 注册一次，通过 LocalActivityResultRegistry

  3. **拆分 SettingsScreen**：把不常用的「重置识别器配置」/「识别反馈管理」抽成二级页面，

  4. **用 LazyColumn 替代 Column**：让卡片懒加载（边际效益不大，因当前内容都在屏内）
  5. **profile-guided 优化**：用 Android Studio Profiler 录 Composition trace，看哪些 Composable 重组耗时长
- **重启条件**：
  - 用户反馈"加载感"明显影响日常使用
  - 或 Sprint 2 完成后剩余 Task 都收完，作为 UI 全打磨阶段的一部分

### [性能] 复习/详情页 KaTeX 渲染时长进一步优化
- **优先级**：P3
- **日期**：2026-04-30
- **背景**：Task 2.1 修复 C 已用 `WebViewClient.onPageFinished` + 遮罩兜住 KaTeX 加载期，

  4. **改用原生 LaTeX 渲染**：JLatexMath 或自研，告别 WebView。但该方向已被项目铁律禁止
     （CLAUDE.md "禁止引入 JLatexMath"）
