# Formula Master · 考研数学公式记忆助手

> 一个纯本地运行的 Android App，用**间隔重复（FSRS）+ 多维卡型 + 认知科学编码仪式**帮考研学生把数学公式真正记牢——不只是背下来，而是"识别 / 填空 / 条件先行 / 推导 / 易混辨析 / 题型反查"六个维度都过关。
>
> 但这个仓库真正想展示的，是另一件事：**它是一个从数据层到 UI 层、从测试到文档，全程由人类主导、AI（Claude Code）全栈协作构建的项目**，并沉淀出一套让"AI 开发"可持续、可回溯、不漂移的**文档驱动方法论**。

---

## 一、项目概况

| | |
|---|---|
| **是什么** | 考研数学公式记忆 App（高数 / 线代 / 概率论），核心功能 **零网络依赖** |
| **给谁** | 考研数学考生（数一 / 数二 / 数三，进 App 自动按子科目过滤公式） |
| **核心引擎** | FSRS 间隔重复算法 + 六类子卡独立记忆状态 + 复习路由器（轮转 / 粘卡 / 加强卡 / 默写） |
| **内容形态** | 每条公式标注 18+ 字段（用途 / 条件 / 推导链 / 典型题 / 易混对 / 口诀 …），LaTeX 由 KaTeX 离线渲染 |
| **设计基线** | 严格 Material 3 + 深色模式 + 严格 MVVM/UDF |

---

## 二、技术栈

- **语言 / UI**：Kotlin + Jetpack Compose（Material 3）
- **架构**：严格 MVVM + UDF 单向数据流；UI 状态封装在 `UiState`，经 ViewModel 的 `StateFlow` 暴露
- **持久化**：Room（SQLite）+ DataStore（偏好，含 Tink AES256-GCM 加密敏感项）
- **异步**：Kotlin Coroutines + Flow
- **公式渲染**：KaTeX 离线包 + WebView（含 WebView 复用池优化）
- **后台**：WorkManager（每日复习提醒）
- **记忆算法**：FSRS（子卡级 stability / difficulty / lapses 独立调度）
- **手写识别**：依赖倒置——Domain 层定义 `MathOcrRecognizer` 接口，UI 层完全不感知具体 SDK

## 三、"全栈 AI" 的两个维度

这个项目对 AI 的态度是**克制而分层**的，刻意区分"哪里该用 AI、哪里坚决不用"：

### 维度一：产品内的 AI —— 用在两端，内核保持确定性

- ✅ **输入端**：手写公式 **OCR 识别**（云端 API，作为*可选增强*通过接口注入；离线仍可用纸笔自评模式）
- ✅ **内容端**：公式库的 18+ 字段标注有一条 **LLM 批量生成 pipeline**（内容工程 Track-B），人工抽检把关
- ⛔ **内核端**：记忆调度与判定**刻意不用大模型**——FSRS、leech 检测、卡型路由全是**确定性算法**。
  这是一条设计铁律：用户的复习节奏必须可解释、可复现，不能交给黑箱。

> 换句话说：AI 负责"看懂输入"和"生产内容"，**不负责决定你今天该复习什么**。

### 维度二：开发中的 AI —— 全栈协作，人类掌舵

从 `FormulaEntity` 的 schema 设计、Room DAO、FSRS 调度器、复习状态机，到 Compose 页面、KaTeX 渲染模板、360+ 单元测试，**每一层都由 Claude Code 协作产出**。
真机验收也走 AI 自动化（`adb` / `uiautomator` 驱动 App + DB 注入构造测试状态 + 截图回读）。

但——**AI 写代码不是难点，让 AI 在几十轮对话、跨越数周、经历上下文丢失后仍然"记得项目在干什么、决策为什么这么定、下一步该做什么"才是难点。**
这正是下面这套文档体系要解决的。

## 四、我的 AI 开发方式 —— 文档驱动方法论 ⭐

AI 每次新对话都是"冷启动"，没有长期记忆。我用一套**分层文档体系**充当项目的"外置大脑"，让任意一次新对话都能在几分钟内接管到正确的上下文，并且**不会偏离既定决策**。

### 四层体系

```
┌─────────────────────────────────────────────────────────────┐
│  ① 开发规则文档   CLAUDE.md                                    │
│     ——「铁律」：AI 每次对话自动加载，定义不可违反的约束         │
│        + 方法论（踩坑换来的强制规则）+ 包结构 + 组件规范        │
├─────────────────────────────────────────────────────────────┤
│  ② 总状态 / 导航   TODO.md（根指针） + 项目记忆库（auto-memory）│
│     ——「我们现在在哪」：指向当前活动文档 + 跨对话进度快照       │
├─────────────────────────────────────────────────────────────┤
│  ③ 阶段 TODO + 阶段总结   docs/phases/0X_*.md + *阶段总结.md    │
│     ——「这个阶段做什么 / 做完了什么」：Sprint 级 Task 与验收    │
├─────────────────────────────────────────────────────────────┤
│  ④ 改进点池   docs/planning/改进点池.md                        │
│     ——「所有新想法的唯一入口」：待评估 / 已纳入 / 已完成 / 已拒绝│
└─────────────────────────────────────────────────────────────┘
```

#### ① 开发规则文档 —— `CLAUDE.md`
项目级"宪法"，AI 每次对话开头自动读取。内容不是文档，而是**约束**：

- **铁律**：禁止在 Composable 里碰数据库；OCR 必须走接口注入；渲染只用 KaTeX 禁止 JLatexMath……
- **方法论**（用真金白银踩坑换来）：涉及外部 SDK/定价必先查官方页不靠记忆；UI bug 修 2 轮没好转立刻改用"诊断驱动"（插运行时数据让用户一张截图定位）；涉及方案选择必须先问人、不替用户拍板。
- **包结构 + M3 组件规范**：让 AI 产出的代码风格与既有代码一致。

#### ② 总状态 / 导航
- **`TODO.md`（仓库根）**：一个**纯指针文件**，不放内容，只回答"当前在哪个阶段、活动文档是哪几个、工作流怎么走"。新对话第一站。
- **项目记忆库（auto-memory，仓库外）**：跨对话的进度快照（`project_progress.md`）+ 索引（`MEMORY.md`）+ 移交指南（累积踩坑清单 + 对话约定）。这是 AI 的"长期记忆"落盘处，随开发滚动更新。

#### ③ 阶段 TODO + 阶段总结
项目按**阶段（Phase）**推进，每阶段拆成若干 **Sprint**，每 Sprint ≤ 8 个 Task：
- **进行中**：`docs/phases/0X_*_TODO.md` —— 每个 Task 带 Done 标准，完成即打勾，并就地记录二级决策与踩坑。
- **已归档**：阶段收尾写一份 `*阶段总结.md`，沉淀"落地了什么 / 留了什么债 / 下一阶段起点"。
- 目前：原型阶段 ✅ → 打磨完善阶段 ✅ → **学习流程重构阶段（进行中）**。

#### ④ 改进点池 —— `docs/planning/改进点池.md`
**所有新想法的唯一集散地**，杜绝"想到什么随手改、打断当前 Sprint"：
- 冒出的点子一律先进 **📥 待评估**（不打断手头工作）；
- 规划新 Sprint 时按 `(优先级 ASC, 提出时间 ASC)` 排序，取 TOP 4–8 移入 **🔄 已纳入 Sprint**；
- Sprint 完成移入 **✅ 已完成**；被否的方案移入 **❌ 已拒绝**（**连同拒绝理由一起留档**，避免以后反复重提同一个被否的点子）。

### 闭环工作流

```
新对话开工  →  读 CLAUDE.md + TODO.md + 当前阶段 TODO（+ 记忆库）
   ↓
开发中冒出新想法  →  写进改进点池「待评估」，继续当前 Task 不打断
   ↓
Task 完成  →  阶段 TODO 打勾 + 记录决策/踩坑
   ↓
Sprint 收尾 / 启动  →  扫改进点池按优先级选 4–8 条 → 移入「已纳入」
   ↓
阶段收尾  →  写阶段总结 → TODO 归档 → 开新阶段
```

### 为什么这套体系有效

- **抗冷启动**：任意新对话 5 分钟接管上下文，不用人重新交代。
- **决策可回溯**：每个"为什么这么定"都在 RFC / 阶段 TODO / 改进点池留痕，包括**被拒绝的方案**——AI 不会好心办坏事重新捡起早被否掉的思路。
- **防漂移**：铁律 + "方案选择先问人"把 AI 约束在既定轨道，避免它自作主张改架构。
- **想法不丢也不乱**：改进点池是唯一入口，既不打断当前 Sprint，又不会让灵感蒸发。

## 六、方法论精华（踩坑换来的铁律）

> 这些不是"最佳实践清单"，是**每一条都对应一次真实翻车**：

- **外部 SDK / 定价，先查官方页，不靠训练记忆**。
  （踩坑：曾靠记忆推荐 ML Kit 手写识别为本地方案 → 实装发现它根本不支持数学公式，浪费一整轮迭代。）
- **UI / 渲染 bug 修 2 轮无果 → 立刻改"诊断驱动"**：在出问题的层插运行时数据（角标 / Logcat），让人一张截图定位根因。
  （踩坑：公式渲染异常连修 5 轮猜根因全错，最后插 `body.clientHeight` 诊断角标一眼看出 CSS `height:100%` 在内嵌 WebView 失效。）
- **涉及方案选择（入口位置 / 默认值 / 交互形式）必须先问人**，给 A/B/C + 利弊，等拍板再动手。
- **真机 DB 调试**：Room 走 WAL，只 pull 主库会读到空表 → 必须 pull `.db + .db-wal + .db-shm` 三件套。
